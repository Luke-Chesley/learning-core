# Pack Gap Taxonomy

Use this when deciding whether to add or expand a pack.

## Add a new pack when
- multiple scenario families want the same interaction style
- current packs cannot express the domain cleanly
- a reusable domain engine or schema would materially improve quality
- generic activity outputs are consistently weak for that domain

## Improve pack detection when
- a relevant pack already exists
- scenarios in that domain are routing to generic activity generation
- lesson content clearly implies the pack but generation misses it

## Improve pack contract when
- the right pack is chosen but produced artifacts are weak or too narrow
- the frontend/runtime can render more than the contract currently allows
- evaluator notes repeatedly mention the same missing field

## Improve upstream lesson/context when
- activity quality is weak because the lesson draft is weak
- pack behavior is reasonable given poor inputs
- activity failures disappear when session generation improves

## Candidate pack signals

### Puzzle / tactics domains
- chess, logic grids, pattern tactics, symbolic reasoning

### Math concept manipulation
- place value, fractions, algebra tiles, geometric decomposition

### Reading / language practice
- phonics manipulation, sentence combining, grammar choice practice, comprehension sequencing

### Science / process domains
- classification, experiment design, variable reasoning, diagram labeling

### History / civics
- timeline sequencing, cause/effect mapping, source comparison, claim/evidence sorting

### Project / maker
- step planning, checklist execution, design tradeoff reflection, observation journaling
